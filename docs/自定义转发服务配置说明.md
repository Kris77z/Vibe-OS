# 自定义转发服务配置说明

## 概述

当前实际使用的模型接入可以收敛为 4 类文本 provider 和 1 类独立图片生成配置：

- `deepseek`: 自定义转发服务，OpenAI-compatible 协议
- `openai`: 自定义转发服务，OpenAI-compatible 协议
- `gemini`: Gemini 转发服务，Google-style `:generateContent` 协议
- `grok`: xAI 官方 API
- `image_gen`: L6 生图专用配置，默认复用 Gemini 凭证

系统的节点级模型配置统一走以下命名规则：

```bash
[NODE_NAME]_LLM_PROVIDER
[NODE_NAME]_LLM_MODEL
[NODE_NAME]_LLM_TEMPERATURE
```

例如：

```bash
L3_AGGREGATE_LLM_PROVIDER=openai
L3_AGGREGATE_LLM_MODEL=gpt-5.1
L3_AGGREGATE_LLM_TEMPERATURE=0.5
```

如果某个节点没有单独配置，会回退到全局 `LLM_PROVIDER`。当前使用口径建议只在以下 provider 中选择：

```text
deepseek / gemini / openai / grok
```

## Provider 与默认配置

下表是当前代码里的实际 provider、环境变量和默认值：

| Provider | 用途 | 认证变量 | Base URL / 端点变量 | 默认模型 | 实际协议 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| `deepseek` | 文本 | `DEEPSEEK_AUTH_KEY` | `DEEPSEEK_BASE_URL`，建议 `https://ai.co.link/deepseek/v1` | `deepseek-chat` | OpenAI-compatible `/chat/completions` |
| `openai` | 文本 | `OPENAI_AUTH_KEY` | `OPENAI_BASE_URL`，建议 `https://ai.co.link/openai/v1` | `gpt-4o` | OpenAI-compatible `/chat/completions` |
| `gemini` | 文本 | `GEMINI_AUTH_KEY` | `GEMINI_BASE_URL`，建议 `https://ai.co.link/gemini-global` | `gemini-2.0-flash` | Google-style `:generateContent` |
| `grok` | 文本 | `GROK_API_KEY`，兼容别名 `XAI_API_KEY` | `GROK_BASE_URL`，默认 `https://api.x.ai/v1` | `grok-4-latest` | xAI OpenAI-compatible `/chat/completions` |
| `image_gen` | 生图 | 默认复用 Gemini key | `IMAGE_GEN_BASE_URL`，默认回退到 `GEMINI_BASE_URL` | `gemini-2.5-flash-image` | Google-style `:generateContent` |

补充说明：

- Gemini 文本在当前使用口径下建议统一走 relay，推荐设置 `GEMINI_AUTH_KEY`。
- Grok 文本调用会优先读取 `GROK_API_KEY`，其次兼容 `XAI_API_KEY`。
- L6 生图默认复用 Gemini 的 key 和 base URL，但模型名独立使用 `IMAGE_GEN_MODEL`。

## 各 Provider 配置说明

### 1. DeepSeek

项目中的 `deepseek` provider 走自定义转发服务，不是官方 DeepSeek 直连。

```bash
DEEPSEEK_AUTH_KEY=your_deepseek_auth_key
DEEPSEEK_BASE_URL=https://ai.co.link/deepseek/v1
DEEPSEEK_MODEL=deepseek-chat
```

说明：

- 协议是 OpenAI-compatible `chat/completions`
- 认证使用 `x-auth-key`
- 代码默认模型是 `deepseek-chat`
- `.env.example` 当前推荐模型是 `deepseek-v3`

### 2. OpenAI / GPT

项目中的 `openai` provider 也走自定义转发服务，不是 OpenAI 官方直连。

```bash
OPENAI_AUTH_KEY=your_openai_auth_key
OPENAI_BASE_URL=https://ai.co.link/openai/v1
OPENAI_MODEL=gpt-4o
```

说明：

- 协议是 OpenAI-compatible `chat/completions`
- 认证依赖 `x-auth-key`
- SDK 层可能附带 `Authorization: Bearer dummy` 占位头，转发服务应忽略它，只看 `x-auth-key`
- 当前代码默认模型是 `gpt-4o`
- 你的实际环境也可以配置为 `gpt-5.1`

`gpt-5.1` 特例：

- 当前代码检测到模型名包含 `gpt-5.1` 时，会主动移除 `temperature`
- 原因是现有转发服务对该模型不接受自定义 `temperature`

示例：

```bash
OPENAI_AUTH_KEY=your_openai_auth_key
OPENAI_BASE_URL=https://ai.co.link/openai/v1
OPENAI_MODEL=gpt-5.1
```

### 3. Gemini 文本

项目中的 `gemini` provider 当前统一按 Gemini relay 使用。

```bash
GEMINI_AUTH_KEY=your_gemini_relay_key
GEMINI_BASE_URL=https://ai.co.link/gemini-global
GEMINI_MODEL=gemini-3-pro-preview
```

说明：

- 当前统一使用 relay 地址，例如 `https://ai.co.link/gemini-global/gemini-3-pro-preview:generateContent`
- Gemini relay 使用 Google 原生 `contents` / `:generateContent` 协议，不是 OpenAI 协议
- 当前文档不再展开官方 Gemini API 配置
- 当前配置建议统一使用 `GEMINI_AUTH_KEY`

### 4. Gemini 生图

L6 生图不走文本 `ChatGoogleGenerativeAI`，而是单独调用图片生成接口。

```bash
IMAGE_GEN_BASE_URL=https://ai.co.link/gemini
IMAGE_GEN_MODEL=gemini-2.5-flash-image
```

说明：

- 如果没有显式设置 `IMAGE_GEN_BASE_URL`，会回退到 `GEMINI_BASE_URL`
- 如果没有显式设置 `IMAGE_GEN_MODEL`，默认是 `gemini-2.5-flash-image`
- 生图请求使用 Google-style `:generateContent`
- 请求体里会使用 `response_modalities=["IMAGE"]`
- 当前代码默认优先复用 Gemini 的 key

### 5. Grok

项目中的 `grok` provider 走 xAI 官方 API。

```bash
GROK_API_KEY=your_grok_api_key
GROK_BASE_URL=https://api.x.ai/v1
GROK_MODEL=grok-4-latest
```

也兼容：

```bash
XAI_API_KEY=your_grok_api_key
```

说明：

- 协议是 OpenAI-compatible `chat/completions`
- 认证使用 `Authorization: Bearer <key>`
- 代码默认模型是 `grok-4-latest`

## 认证与请求结构

### DeepSeek relay

```bash
curl -X POST https://ai.co.link/deepseek/v1/chat/completions \
  -H "Content-Type: application/json" \
  -H "x-auth-key: your_auth_key" \
  -d '{
    "model": "deepseek-v3",
    "messages": [{"role": "user", "content": "Hello"}]
  }'
```

### OpenAI relay

```bash
curl -X POST https://ai.co.link/openai/v1/chat/completions \
  -H "Content-Type: application/json" \
  -H "x-auth-key: your_auth_key" \
  -H "Authorization: Bearer dummy" \
  -d '{
    "model": "gpt-4o",
    "messages": [{"role": "user", "content": "Hello"}],
    "stream": true
  }'
```

说明：

- `Authorization: Bearer dummy` 是 SDK 兼容层占位，不是实际认证凭证
- 真正认证字段是 `x-auth-key`

### Gemini relay

```bash
curl -X POST https://ai.co.link/gemini-global/gemini-3-pro-preview:generateContent \
  -H "Content-Type: application/json" \
  -H "x-auth-key: your_auth_key" \
  -d '{
    "contents": [
      {
        "role": "user",
        "parts": [{"text": "Hello"}]
      }
    ],
    "generationConfig": {
      "temperature": 0.3
    }
  }'
```

说明：

- Gemini relay 不使用 `messages`
- 它使用 Google 原生的 `contents` 格式

### Grok 官方 API

```bash
curl https://api.x.ai/v1/chat/completions \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer your_api_key" \
  -d '{
    "model": "grok-4-latest",
    "messages": [{"role": "user", "content": "Hello"}]
  }'
```

### L6 生图接口

```bash
curl -X POST https://ai.co.link/gemini/gemini-2.5-flash-image:generateContent \
  -H "Content-Type: application/json" \
  -H "x-auth-key: your_auth_key" \
  -d '{
    "contents": [
      {
        "role": "user",
        "parts": [{"text": "A simple blue square on white background"}]
      }
    ],
    "generationConfig": {
      "response_modalities": ["IMAGE"],
      "image_config": {
        "aspect_ratio": "16:9"
      }
    }
  }'
```

## 节点级模型配置

`.env.example` 当前给出了以下节点模板：

```bash
# L2 / L3 / L4
L2_EXTRACT_LLM_PROVIDER=gemini
L2_EXTRACT_LLM_MODEL=gemini-3-pro-preview
L3_AGGREGATE_LLM_PROVIDER=gemini
L3_AGGREGATE_LLM_MODEL=gemini-3-pro-preview
L4_DECISION_LLM_PROVIDER=gemini
L4_DECISION_LLM_MODEL=gemini-3-pro-preview

# L5.1 采编分离
L5_VERIFIER_LLM_PROVIDER=openai
L5_VERIFIER_LLM_MODEL=gpt-4o
L5_BRIEF_LLM_PROVIDER=openai
L5_BRIEF_LLM_MODEL=gpt-4o
L5_SYNTHESIZER_LLM_PROVIDER=grok
L5_SYNTHESIZER_LLM_MODEL=grok-4-latest

# L5.2 / L5.3 / L5.4
L5_WRITER_LLM_PROVIDER=gemini
L5_WRITER_LLM_MODEL=gemini-3-pro-preview
L5_PLANNER_LLM_PROVIDER=openai
L5_PLANNER_LLM_MODEL=gpt-4o
L5_COMPOSER_LLM_PROVIDER=openai
L5_COMPOSER_LLM_MODEL=gpt-4o
L5_EDITOR_LLM_PROVIDER=gemini
L5_EDITOR_LLM_MODEL=gemini-3-pro-preview
L5_FORMAT_VALIDATOR_LLM_PROVIDER=gemini
L5_FORMAT_VALIDATOR_LLM_MODEL=gemini-3-pro-preview

# L6
L6_VISUAL_LLM_PROVIDER=gemini
L6_VISUAL_LLM_MODEL=gemini-3-pro-preview
IMAGE_GEN_MODEL=gemini-2.5-flash-image

# 其他节点
HISTORY_SUMMARIZER_LLM_PROVIDER=deepseek
HISTORY_SUMMARIZER_LLM_MODEL=deepseek-v3
FLASH_L4_ADAPTER_LLM_PROVIDER=deepseek
FLASH_L4_ADAPTER_LLM_MODEL=deepseek-v3
```

代码中还额外使用了以下节点名，但 `.env.example` 当前没有预置模板：

- `L5_SCORER`
- `L6_TWITTER`

如果需要给它们单独指定模型，也可以直接按同样规则配置：

```bash
L5_SCORER_LLM_PROVIDER=deepseek
L5_SCORER_LLM_MODEL=deepseek-v3

L6_TWITTER_LLM_PROVIDER=openai
L6_TWITTER_LLM_MODEL=gpt-4o
```

## 推荐配置示例

下面这份示例与当前 `.env.example` 的思路一致，适合作为生产起点：

```bash
# 全局默认 provider
LLM_PROVIDER=gemini

# DeepSeek relay
DEEPSEEK_AUTH_KEY=your_deepseek_auth_key
DEEPSEEK_BASE_URL=https://ai.co.link/deepseek/v1
DEEPSEEK_MODEL=deepseek-v3

# OpenAI relay
OPENAI_AUTH_KEY=your_openai_auth_key
OPENAI_BASE_URL=https://ai.co.link/openai/v1
OPENAI_MODEL=gpt-5.1

# Gemini relay
GEMINI_AUTH_KEY=your_gemini_relay_key
GEMINI_BASE_URL=https://ai.co.link/gemini-global
GEMINI_MODEL=gemini-3-pro-preview

# Grok
GROK_API_KEY=your_grok_api_key
GROK_BASE_URL=https://api.x.ai/v1
GROK_MODEL=grok-4-latest

# L6 生图
IMAGE_GEN_BASE_URL=https://ai.co.link/gemini
IMAGE_GEN_MODEL=gemini-2.5-flash-image
```

## 验证与测试

当前仓库里和模型配置直接相关的验证脚本如下：

### 1. 测试节点级配置解析

```bash
./venv/bin/python tests/unit/test_config.py
```

这个脚本会检查：

- 全局 provider 选择
- 节点级 provider / model / temperature 覆盖
- 指定节点的客户端初始化

### 2. 测试各 relay / API 的可用模型

```bash
./venv/bin/python tests/integration/test_available_models.py
```

这个脚本当前覆盖：

- DeepSeek relay
- OpenAI relay
- Gemini relay
- Grok 官方 API

注意：

- 它不覆盖 L6 图片生成

### 3. 测试 L6 生图 relay

```bash
./venv/bin/python tests/test_image_gen_response.py
```

该脚本会检查：

- `IMAGE_GEN_BASE_URL`
- `IMAGE_GEN_MODEL`
- Gemini key 是否可用
- 生图接口的响应结构和 `usageMetadata`
